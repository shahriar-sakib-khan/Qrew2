# Qrew Application Architecture & AI Agent Instructions

## 1. Project Overview & Tech Stack
You are operating within "Qrew", an Enterprise SaaS for SME Office Management in Bangladesh.
This is a Monorepo workspace (Turborepo + pnpm) separated into two primary applications:
*   **Frontend (`apps/web`):** Next.js 15 (App Router), React 19, Tailwind CSS v4, Shadcn/UI, React Hook Form, Zod.
*   **Backend (`apps/api`):** Hono.js, Node.js, Better Auth, PostgreSQL (Drizzle ORM), Redis, BullMQ.
*   **Infrastructure:** Docker Compose (local Postgres/Redis), Cloudflare R2 (Storage), Resend (Email).

---

## 2. Core Architectural Principles (Separation of Concerns)
*   **Strict Frontend/Backend Boundary:** The Next.js frontend MUST NEVER execute direct database queries. All database operations, business logic, and heavy processing must reside in the Hono API (`apps/api`). The frontend acts strictly as a UI consumption layer.
*   **Smart Container / Dumb Component Pattern:** Complex React views MUST lift state to a parent "Container" component. The parent acts as the Orchestrator, holding the `react-hook-form` `<FormProvider>` context and executing the single `onSubmit` function. Child components (like inputs or uploaders) must be purely presentational.
*   **Backend Folder Structure:** The `apps/api` logic MUST be strictly separated into `routes/` (endpoint definitions), `controllers/` (request handling), and `services/` (business logic & DB calls).

---

## 3. UI/UX & Accessibility Standards
*   **Native Accessibility Focus Rings:** NEVER use Tailwind's `ring` utilities for focus states due to SPA mouse-click retention bugs. System-wide, buttons and links MUST use native CSS outlines: `focus-visible:outline focus-visible:outline-1 focus-visible:outline-foreground/15 focus-visible:outline-offset-0`.
*   **Dirty State Highlighting:** Unsaved form fields MUST provide immediate visual feedback. If `formState.isDirty` is true, the input receives a specific visual cue (e.g., emerald tinted border/background).
*   **The Unsaved Changes Guard:** Unsaved forms must be protected by a `beforeunload` window event listener AND a fixed, high-visibility (`z-50`) floating banner offering "Save" or "Discard" actions.
*   **Layout & Scrolling Physics:**
    *   To prevent the "Flexbox Min-Height Bug", any `<ScrollArea>` used alongside `flex-1` MUST include the `min-h-0` utility class.
    *   Never mix native `overflow-y-auto` inside a Radix `<ScrollArea>`.
    *   Nested layouts (like Settings) must use negative margins (e.g., `-m-4 lg:-m-6`) to sit flush against main App Shell boundaries.
*   **Hydration Guards:** Any state derived from `localStorage` (like Collapsible Sidebars) must be wrapped in a `useEffect` mount check, returning `null` or a skeleton until `isMounted` is true to prevent SSR/CSR hydration crashes.

---

## 4. Cybersecurity & Data Integrity (Bulletproof Code)
*   **Zod Sanitization & Validation:** ALL incoming API requests to Hono MUST be strictly validated and sanitized using `@hono/zod-validator`. Never trust client payload data.
*   **Global Rate Limiting:** All API endpoints MUST be protected by a global Redis-backed Rate Limiter middleware to prevent DDoS and Brute Force attacks.
*   **Content Security Policy (CSP):** The Next.js `next.config.mjs` MUST enforce strict CSP headers, preventing unauthorized iframe embedding (Clickjacking), XSS, and restricting image/script sources.
*   **Secure File Handling:**
    *   Avatars/Images must be compressed client-side via `browser-image-compression` BEFORE network transmission.
    *   Files MUST be uploaded directly to Cloudflare R2 using short-lived Presigned POST URLs generated by the backend. The backend must cryptographically enforce `maxSize` and `contentType` during URL generation.
*   **Database Hooks:** Security metadata (like `last_login_at`) must be stamped into the Postgres DB via Better Auth's `databaseHooks.session.create.after` lifecycle hooks, as Redis alone handles session checks.

---

## 5. Observability & Industry Standards
*   **Industry Standard Logging:** Use `pino` (structured JSON logging) across the backend. DO NOT use raw `console.log()` in production API code. Log critical events, auth failures, and external API latencies.
*   **Error Tracking (Sentry):** Implement Sentry boundaries in both the Hono backend and Next.js frontend (`error.tsx` and `global-error.tsx`). Handle React crashes gracefully without exposing stack traces to the user.

---

## 6. Testing Protocol
*   **Backend Testing (Vitest + Supertest):** Every new Hono route or controller MUST have a corresponding test file created in `apps/api/__tests__`. Tests must verify 200 OKs, 400 Validation Errors, and 401/403 Auth failures.
*   **Frontend E2E (Playwright):** Critical user journeys (Sign In, Profile Update, Role-based Routing checks) must be covered by Playwright tests in `apps/web`.

---

## 7. DevOps & Infrastructure Rules
*   **Safe Database Resets:** NEVER write destructive raw SQL commands (like `DROP SCHEMA`) into `package.json` scripts.
*   **Container Wiping:** To wipe the local database, use the strict Docker volume command: `docker compose down -v --remove-orphans && docker compose up -d && sleep 3 && pnpm db:push`.
*   **IPv6 Resolution:** When connecting Next.js/Hono to local Docker containers, strictly use `127.0.0.1` instead of `localhost` in `.env` files to prevent Node v17+ IPv6 resolution errors.
*   **Memory Management:** Always call `URL.revokeObjectURL()` in a `useEffect` cleanup function when staging local file uploads to prevent browser memory leaks.
